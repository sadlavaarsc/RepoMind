# RepoMind - RAG System Development Plan

## **🎯 Project Goal**

Build a **code-aware Retrieval-Augmented Generation (RAG) system** for repository understanding, with:

- Strong retrieval performance (measurable via Recall@k, MRR)
- Lightweight but extensible architecture
- API-based access (production-oriented design)
- Clear separation between ingestion, retrieval, and generation

This is a **demo-first implementation**, but must reserve extensibility for future features.

# **🧠 1. Overall Architecture**

Implement the following modular structure:

```
repomind/
│
├── ingestion/        # data parsing and preprocessing
├── indexing/         # embedding + vector index
├── retrieval/        # retrieval pipeline (core logic)
├── generation/       # LLM-based answer generation
├── evaluation/       # metrics and benchmarking
├── api/              # FastAPI service
├── storage/          # vector store abstraction
└── configs/
```



# **⚙️ 2. Technology Selection & Constraints**

## **2.1 Vector Store**





- Use **FAISS** (required)
- DO NOT use external vector DB (e.g., Qdrant) in demo phase
- MUST implement a **VectorStore abstraction layer**:



```
class VectorStore:
    def add(documents): ...
    def search(query, top_k): ...
```



- Design so that future migration to Qdrant is possible





------





## **2.2 Embedding**





- Use pre-trained embedding model (e.g., BGE-small or similar)
- DO NOT train embedding model
- Wrap embedding in a service class:



```
class EmbeddingService:
    def embed(text): ...
```



------





## **2.3 LLM Usage**





- Use LLM ONLY for:

  

  - Query expansion (MQE)
  - Final answer generation

  

- DO NOT use LLM in ingestion phase (for demo)





------





## **2.4 Framework Constraints**





- DO NOT use heavy frameworks like LangChain

- Implement pipeline manually (required)

- Use:

  

  - Python
  - FastAPI
  - asyncio (optional for concurrency)

  





------





# **🧱 3. Ingestion (Preprocessing) Design**







## **✅ MUST implement (core)**







### **3.1 Code Chunking**





- Parse repository into **function-level chunks**

- For Python:

  

  - Use ast module

  

- For other languages:

  

  - Use simple heuristic (regex or file-level split)

  





------





### **3.2 Metadata Extraction**





Each chunk MUST include:

```
{
  "content": "...",
  "file_path": "...",
  "function_name": "...",
  "class_name": "...",
  "language": "..."
}
```



------





## **❌ DO NOT implement (for now)**





- LLM-based summarization
- Graph-based code analysis
- Multi-modal parsing





------





## **🟡 RESERVED (future extension)**





Leave hooks for:

```
def summarize_chunk(chunk): ...
```



------





# **🧠 4. Retrieval Pipeline (CORE MODULE)**





Implement a **multi-stage retrieval pipeline**:



------





## **Step 1: Query Expansion (MQE)**





- Generate 2–3 query variants
- Use LLM OR simple template expansion





------





## **Step 2: Vector Search**





- Use FAISS
- Retrieve top_k = 20





------





## **Step 3: Metadata Filtering (MUST)**





- Implement manual filtering logic:

  

  - file extension (e.g., .py)
  - keyword match in file_path

  

- This compensates for FAISS limitations





------





## **Step 4: Reranking (MUST)**





Implement lightweight scoring:

```
score = α * embedding_score + β * keyword_overlap
```



- No cross-encoder
- No heavy models





------





## **Step 5: Context Packing**





- Deduplicate chunks
- Limit total token size
- Preserve source references





------





# **🤖 5. Generation Module**







## **MUST implement:**





- Prompt template:



```
Answer the question based on the following code context.
Cite file paths when possible.
```



- Include retrieved chunks as context

- Output must include:

  

  - Answer
  - References (file paths)

  





------





## **RESERVED:**





- hallucination detection
- answer verification





------





# **📊 6. Evaluation System (REQUIRED)**





Implement evaluation scripts:



------





## **Metrics:**







### **6.1 Recall@k**





- Whether correct file appears in top-k







### **6.2 MRR**





- Rank of first correct result







### **6.3 Latency**





- End-to-end query time







### **6.4 Token Usage**





- Prompt + completion tokens





------

```
query → expected file/function
```
